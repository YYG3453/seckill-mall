# 秒杀商城项目面试题（含口语化答案 + 八股要点）

> 适用范围：本项目后端（Spring Boot + MyBatis-Plus + MySQL + Redis + Lua + Session）与前端（Vue3 + Pinia + Router + Axios + Element Plus + ECharts + Vite）  
> 使用方式：先看“口语版答案”应对面试，再用“八股要点”补全细节。

---

## 1. 项目整体与架构

### Q1：你这个项目是干什么的？核心难点是什么？
**口语版答案**  
这是一个秒杀商城，普通电商功能（商品、购物车、下单、支付模拟、后台管理）+ 秒杀高并发链路。难点主要是秒杀场景下防超卖、防重复下单、抗刷、以及 Redis 和 MySQL 的一致性补偿。

**八股要点**  
- 业务域：用户、商品、订单、秒杀场次、通知、推荐、监控  
- 秒杀链路：动态路径 + 限流 + Redis Lua 原子扣减 + DB 条件更新兜底 + 失败补偿  
- 非功能：可观测（监控任务、异常日志）、可维护（分层清晰）

**项目落地**  
- `SeckillController` + `SeckillService` + `SeckillTxService` + `lua/seckill.lua`

---

### Q2：你为什么是前后端分离？好处是什么？
**口语版答案**  
前后端分离能让页面开发和接口开发并行，职责更清楚。前端专注交互，后端专注业务与数据，后续换客户端也更方便。

**八股要点**  
- 解耦：UI 与业务解耦  
- 并行开发：接口契约优先  
- 可扩展：Web/小程序/App 复用后端接口  
- 统一接口返回结构：`ApiResult`

---

### Q3：为什么要分 controller / service / mapper？
**口语版答案**  
Controller 处理 HTTP，Service 放业务规则，Mapper 只负责数据库访问。这样改业务不会牵扯太多接口层，查问题也更快。

**八股要点**  
- 单一职责  
- 可测试性  
- 可维护性与复用性  
- 避免“胖 Controller”

---

## 2. Java 基础与并发

### Q4：`synchronized` 在你的项目里用在哪，为什么？
**口语版答案**  
订单号生成用了 `synchronized`，因为生成逻辑要保证同一时刻线程安全，不能重复号。

**八股要点**  
- 互斥 + 可见性 + 有序性  
- 适合临界区短、竞争可控场景  
- 结合项目：`OrderNoGenerator#next()`

---

### Q5：`AtomicLong` 和 `synchronized` 怎么选？
**口语版答案**  
简单计数器优先 `AtomicLong`，复杂多步骤一致性还是 `synchronized` 或锁。项目里两个都用了：计数场景用 Atomic，订单号生成是多步骤所以同步方法。

**八股要点**  
- `Atomic*`：CAS 无锁，单变量原子操作  
- `synchronized`：代码块级别互斥，适合复合操作  
- 组合使用很常见

---

### Q6：为什么时间处理要注意格式？
**口语版答案**  
前后端时间格式不一致会出错，尤其秒杀开始/结束时间。我们统一了格式，并在前端对日期字符串做了兼容处理。

**八股要点**  
- ISO 8601、时区  
- Jackson 时间序列化配置  
- 前端 date picker 的 `value-format`

---

## 3. Spring Boot / MVC / AOP / Interceptor

### Q7：`@RestController` 和 `@Controller` 区别？
**口语版答案**  
`@RestController` 默认返回 JSON，适合前后端分离；`@Controller` 通常配模板渲染页面。

**八股要点**  
- `@RestController = @Controller + @ResponseBody`

---

### Q8：拦截器和 AOP 有什么区别？
**口语版答案**  
拦截器偏 Web 层，按 URL 拦请求；AOP 偏方法切面，按注解或表达式切。项目里登录权限用拦截器，限流和日志用 AOP。

**八股要点**  
- Interceptor：`HandlerInterceptor`，请求生命周期  
- AOP：`@Aspect`，方法级横切  
- 组合使用：鉴权 + 限流 + 日志

**项目落地**  
- `LoginInterceptor`、`RateLimitAspect`、`LogAspect`

---

### Q9：你是怎么做登录鉴权的？
**口语版答案**  
登录成功把 `SessionUser` 放进 Session，拦截器检查是否登录。访问 `/api/admin/**` 还要检查 `role=admin`。

**八股要点**  
- Session 鉴权  
- 角色权限校验  
- 白名单放行路径（注册、登录、公开商品、公开秒杀等）

---

### Q10：全局异常处理有什么好处？
**口语版答案**  
统一错误返回，前端不用每个接口写一套错误格式处理；还能把数据库、Redis 连接问题给出更明确提示，排障更快。

**八股要点**  
- `@RestControllerAdvice + @ExceptionHandler`  
- 业务异常 vs 系统异常分层  
- 可观测性与用户体验提升

---

## 4. Spring Session + Redis

### Q11：为什么登录态放 Redis？
**口语版答案**  
Session 放 Redis 后，多实例部署也能共享登录态，不会出现“切节点就掉登录”的问题。

**八股要点**  
- 无状态扩容友好  
- 跨实例 Session 共享  
- 代价：依赖 Redis 可用性

---

### Q12：Session 鉴权和 JWT 怎么选？
**口语版答案**  
这个项目偏教学与后台管理，Session 更直接；如果是纯网关、多端强扩展，JWT 更常见。两者没有绝对优劣，主要看场景。

**八股要点**  
- Session：服务端可控、可强制失效  
- JWT：无状态、跨服务方便、撤销较难  
- 混合方案也常见（短 JWT + 刷新机制）

---

## 5. MyBatis-Plus / ORM

### Q13：你为什么用了 MyBatis-Plus？
**口语版答案**  
常规 CRUD 和分页很省代码，业务复杂的地方再写自定义 SQL，开发效率高。

**八股要点**  
- BaseMapper  
- LambdaQueryWrapper  
- 分页拦截器

---

### Q14：你遇到过 SQL 条件优先级问题吗？
**口语版答案**  
遇到过。商品搜索里 `AND + OR` 组合如果不加括号会把状态条件冲掉。后来用 `nested` 把 OR 组包起来，结果才正确。

**八股要点**  
- SQL 运算优先级  
- OR 条件分组  
- 框架链式 API 最终仍是 SQL

---

## 6. MySQL 基础题（本项目高频）

### Q15：订单和订单明细为什么拆表？
**口语版答案**  
一单可能多商品，拆成主表 + 明细是标准做法，统计和扩展都更清楚。

**八股要点**  
- 一对多建模  
- 订单状态在主表，商品行在子表  
- 便于索引与统计

---

### Q16：你们有哪些关键索引？
**口语版答案**  
像用户名唯一索引、订单号唯一索引、订单状态+时间索引、用户+已读索引等，都是围绕查询路径设计的。

**八股要点**  
- `uk_username`、`uk_order_no`  
- `idx_status_time`、`idx_user_read`  
- 索引要和 where/order by 场景匹配

---

### Q17：事务在项目里用在哪？
**口语版答案**  
下单、支付状态变更、超时取消这些都要事务，确保“写订单 + 扣库存/回库存”一致。

**八股要点**  
- `@Transactional`  
- 回滚边界  
- 业务补偿与事务配合

---

## 7. Redis / Lua / 秒杀高并发

### Q18：为什么秒杀要用 Lua，不直接 Java 里先查再扣？
**口语版答案**  
先查再扣在高并发下会超卖。Lua 在 Redis 单线程里原子执行，能把“判断 + 扣减 + 限购标记”一次做完。

**八股要点**  
- Redis 单线程 + Lua 原子性  
- 减少网络往返  
- 防超卖、防重复抢购

---

### Q19：你怎么防止一个用户重复抢同一场？
**口语版答案**  
Lua 里用 `SET NX EX` 写用户限购 key，写成功才算占坑；写失败就回滚库存并返回失败。

**八股要点**  
- key 设计：`seckill:user:item:{userId}:{itemId}`  
- `NX` 防重复，`EX` 防脏 key 长期堆积  

---

### Q20：Redis 扣成功但 DB 扣失败怎么办？
**口语版答案**  
做补偿：Redis 库存 `INCR` 回去，限购 key 删掉，让数据回到扣减前状态。

**八股要点**  
- 最终一致性思路  
- 失败补偿而非盲目重试  
- 日志/监控必不可少

---

### Q21：为什么还要数据库条件扣库存，Lua 不是已经扣了吗？
**口语版答案**  
Redis 更像高并发闸门，MySQL 才是最终账本。DB 条件更新是最后一道兜底，防止 Redis/DB 偏差扩大。

**八股要点**  
- Redis 快、DB 准  
- 双层防线  
- 对账与修复

---

### Q22：动态路径是干嘛的？
**口语版答案**  
避免固定秒杀 URL 被别人脚本直接狂刷。用户必须先拿短时 path，再带 path 下单。

**八股要点**  
- `path` + TTL  
- 防接口枚举、防脚本复用

---

### Q23：限流怎么做的？
**口语版答案**  
`@RateLimit` + AOP，在 Redis 里按用户和 IP 两套 key 计数，超过阈值直接拦。

**八股要点**  
- 固定窗口算法  
- `INCR` + `EXPIRE`  
- 用户维 + IP 维组合

---

### Q24：你有做监控吗？
**口语版答案**  
有。按分钟统计请求量和异常，定时检查 QPS 和库存差异，超过阈值写异常日志。

**八股要点**  
- 指标：QPS、限流次数、库存偏差、Lua 连续失败  
- 落库：`exception_log`

---

## 8. 文件上传与静态资源

### Q25：上传图片怎么做的？
**口语版答案**  
后端校验大小和类型，存到本地目录，返回 `/uploads/...` 路径；Spring MVC 把这个路径映射成可访问静态资源。

**八股要点**  
- `MultipartFile`  
- 白名单校验（JPG/PNG/GIF/WEBP）  
- 路径安全（防路径穿越）

## 9. AI 集成（OpenAI 兼容接口）

### Q26：AI 是怎么接入的？
**口语版答案**  
后端通过 `RestTemplate` 调兼容 OpenAI 的 `/chat/completions`，优先走大模型，失败就回退本地规则库。

**八股要点**  
- 配置化开关与 key  
- system prompt + history  
- 降级策略（fallback）

---

### Q27：怎么让 AI 回答“当前账号实时数据”？
**口语版答案**  
每次聊天请求都临时查库拼实时快照到 system hint。换账号后下一次提问自动变成新账号数据；管理员会额外看到全站汇总。

**八股要点**  
- Request-time data hydration  
- RBAC 视图差异（user vs admin）  
- 隐私边界（聚合数据优先）

---

## 10. Vue3 / Pinia / Router / Axios

### Q28：`script setup` 的好处？
**口语版答案**  
语法更简洁、类型推导更自然，写起来比 Options API 更快，组件逻辑集中。

**八股要点**  
- 编译时宏  
- 更少样板  
- 组合式 API 友好

---

### Q29：Pinia 在这个项目干了什么？
**口语版答案**  
存全局用户态、购物车角标、AI 面板开关。路由守卫会先 `loadMe` 再判断能不能进页面。

**八股要点**  
- 全局状态集中管理  
- 与 router guard 协同  
- 与接口层解耦

---

### Q30：路由守卫里做了哪些事？
**口语版答案**  
先恢复登录态，再做权限判断：要登录的页面拦截，后台页面要求 admin。管理员还会被限制只走 `/admin`。

**八股要点**  
- `beforeEach`  
- `meta.auth` / `meta.admin`  
- 重定向与安全校验

---

### Q31：Axios 统一封装有什么价值？
**口语版答案**  
统一错误提示、统一 code 处理、统一带 cookie，避免每个页面都写重复逻辑。

**八股要点**  
- request/response interceptors  
- 401 统一清理用户态  
- FormData 处理 Content-Type

---

### Q32：为什么你要 `withCredentials: true`？
**口语版答案**  
因为登录态走 Session + Cookie，不带 cookie 后端就识别不到用户。

**八股要点**  
- 跨域带凭证  
- 后端 CORS `allowCredentials(true)` 必须配套

## 12. 秒杀专题高频追问（重点）

### Q35：秒杀链路完整时序怎么说？
**口语版答案**  
用户先拿动态 path -> 带 path 下单 -> AOP 限流 -> Lua 原子扣库存 + 限购 -> DB 事务写订单并条件扣库存 -> 成功返回订单号，失败做补偿。

**八股要点**  
- 防刷（path + 限流）  
- 防超卖（Lua + DB）  
- 一致性（补偿）

---

### Q36：超时未支付为什么要定时取消？
**口语版答案**  
不取消会长期占库存，影响真实可售。定时任务把超时订单改已取消并回补库存。

**八股要点**  
- 定时任务扫描  
- 普通库存与秒杀库存分开回补  
- 幂等与重复执行安全

---

### Q37：你如何避免“搜索一输入关键词就全量返回”这类 bug？
**口语版答案**  
核心是 SQL 条件组合正确，尤其 `AND` 和 `OR` 的优先级。项目里用 nested 显式分组解决了这个坑。

**八股要点**  
- 逻辑分组  
- 单测/接口测试覆盖关键条件

---

## 13. 安全与工程规范

### Q38：项目有哪些安全点？
**口语版答案**  
有登录拦截、角色校验、限流、动态路径、防重复抢购、上传白名单、统一异常。还需要继续加强敏感配置管理。

**八股要点**  
- 认证鉴权  
- 输入校验  
- 资源访问控制  
- 审计日志

---

### Q39：配置里的密钥怎么管理更规范？
**口语版答案**  
不要把真实 key 写进仓库，应该走环境变量、密钥服务或 CI 注入。生产环境要定期轮换。

**八股要点**  
- `application.yml` 仅保留占位  
- `.env` / Secret Manager  
- 最小权限 + 轮换机制

---

### Q40：你会怎么做接口测试和压测？
**口语版答案**  
接口测试先覆盖登录、下单、秒杀边界；压测重点打秒杀接口，观察库存一致性、错误率和响应时间，再调限流和 Redis 参数。

**八股要点**  
- 功能测试 + 并发压测  
- 指标：TPS、P99、错误率、超卖率  
- 监控闭环：发现问题 -> 定位 -> 修复 -> 回归

## 14. 前后端调用关系速记（面试表达模板）

### Q41：你怎么快速讲清“谁调用谁”？
**口语版答案模板**  
我一般按“页面动作 -> api 函数 -> controller 方法 -> service 业务”的顺序讲。  
比如购物车结算：`Cart.vue` 点击结算 -> `api/order.createOrderFromCart` -> `OrderController#fromCart` -> `OrderService#createFromCart`。

---

### Q42：你如何保证代码可读性，方便新人接手？
**口语版答案**  
我做了三层文档化：  
1）文件级注释（这个文件做什么）  
2）调用链注释（谁调用谁）  
3）`docs/CALL_FLOW.md` 总览图。  
新人先看总览，再看 API 和 Controller，学习路径很顺。

## 18. 高频追问“反杀版”答案（可直接背）

### 18.1 「如果 Redis 挂了怎么办？」
**标准回答**  
先保证“降级可用”，再保证“数据安全”。  
1) 登录态依赖 Redis 时，先切维护提示或只读模式；  
2) 秒杀入口临时关闭（避免乱单）；  
3) 普通商品浏览走 DB 兜底；  
4) Redis 恢复后做库存与订单对账。  
项目里我们已经有 Redis 异常统一处理和监控日志，后续可加开关做自动熔断。

---

### 18.2 「如果 MySQL 慢查询很多怎么办？」
**标准回答**  
我会按“先定位再优化”做：  
1) 开慢 SQL 日志，定位 TopN；  
2) 看执行计划，补组合索引；  
3) 把高频读热点前置到 Redis；  
4) 必要时分页/限流/异步化。  
这个项目里订单、用户通知、商品查询都能按访问路径继续优化索引和缓存策略。

---

### 18.3 「为什么不用消息队列削峰？」
**标准回答**  
当前体量下，Lua + 限流 + 短链路补偿已经能稳定跑，先保证方案简单可控。  
如果峰值再上一个数量级，我会加 MQ，把“下单确认/通知/日志”等异步化，核心扣库存仍保持强约束。

---

### 18.4 「你如何证明不会超卖？」
**标准回答**  
不是靠“口头保证”，而是三层防线：  
1) Lua 原子扣减防并发读写竞争；  
2) DB 条件更新兜底；  
3) DB 失败时 Redis 回滚补偿。  
另外配库存偏差监控与对账日志，出现偏差能发现和修复。

---

### 18.5 「为什么管理员不能进商城前台？」
**标准回答**  
这是业务约束，不是技术限制。避免管理员用运营账号参与前台购买造成数据噪声。  
所以路由守卫里做了“管理员统一走 `/admin`”的限制，后台和商城角色边界更清晰。

---

### 18.6 「你这套限流是滑动窗口吗？」
**标准回答**  
当前是固定窗口（`INCR + EXPIRE`），实现简单成本低。  
如果要更平滑可升级滑动窗口/令牌桶，比如 Redis sorted set 或网关层限流。

---

### 18.7 「AI 回答怎么保证不乱说？」
**标准回答**  
做了三件事：  
1) 每次请求动态注入“当前账号实时快照”；  
2) system prompt 明确要求优先依据快照回答；  
3) 大模型失败时回退本地规则。  
所以在订单数量、用户量、库存类问题上会更稳。

---

### 18.8 「如果用户投诉‘我明明抢到了却没单’？」
**标准回答**  
先查日志链路：限流命中、Lua返回、事务是否回滚、补偿是否执行。  
如果是 DB 失败触发补偿，会明确给用户失败结果且库存回补，不会出现“成功提示但无单”的静默错误。  
后续可加请求流水号和链路追踪提升定位速度。

---

### 18.9 「你如何做接口幂等？」
**标准回答**  
秒杀天然做了“用户-场次唯一标记”；  
支付/发货这种状态流转接口，要加状态前置校验（待支付才能支付，已支付才能发货）。  
必要时可加幂等 token 防重复提交。

---

引入动态路径与双维限流抵御刷单；
