## 二、8分钟项目讲解（主线版）

### 1) 项目背景（30秒）
这是一个秒杀电商演示项目，既有普通商城流程（商品、购物车、下单、支付模拟），也有高并发秒杀场景。目标是验证在高并发下的数据一致性和系统稳定性。

### 2) 架构与分层（1分钟）
- 前端是 Vue3，页面通过 `api/*.js` 调后端。  
- 后端按 `controller -> service -> mapper` 分层。  
- 登录态走 Spring Session + Redis。  
- 秒杀热点走 Redis + Lua，最终账本在 MySQL。  
- 监控通过定时任务和异常日志落库做可观测。

### 3) 秒杀核心链路（3分钟，重点）
用户点击抢购时流程是：
1. 先请求动态 path，防止固定 URL 被脚本直接刷。  
2. 提交秒杀请求进入 AOP 限流（用户维 + IP 维）。  
3. 通过 Lua 在 Redis 原子执行：  
   - 查库存  
   - 校验用户是否抢过  
   - 扣库存  
   - 写限购标记  
4. Lua 成功后进入 DB 事务：写订单、写明细、条件扣 DB 库存。  
5. 如果 DB 扣减失败，做补偿：Redis 库存加回、限购 key 删除。  

这套方案核心是“三层防线”：  
**限流防刷 -> Lua 防并发超卖 -> DB 兜底 + 补偿保一致性**。

### 4) 登录与权限（1分钟）
- 登录成功后把 `SessionUser` 存 Session。  
- 拦截器统一校验登录；访问 `/api/admin/**` 必须 `role=admin`。  
- 前端路由守卫会先 `loadMe` 恢复会话，再判断 `auth/admin`。  
- 管理员被限制走后台路由，避免角色混用导致业务噪声。

### 5) 可观测与运维友好（1分钟）
- 记录分钟级指标：请求数、限流次数、异常次数。  
- 定时检查 QPS 和库存偏差，超过阈值写异常日志。  
- 全局异常统一返回，数据库/Redis故障有明确提示，便于快速排障。

### 6) AI 能力（45秒）
AI 对话不是纯静态问答。每次请求都会查当前账号实时数据，拼到系统提示里。  
普通用户只看本人数据，管理员额外看全站汇总。模型不可用时回退本地规则，保证可用性。

### 7) 结尾（45秒）
这个项目我最看重的是“可解释性”：每个技术选择都能说明白为什么。  
下一步我会补两件事：  
1) 压测报告（TPS、P99、错误率、超卖率）；  
2) Prometheus + Grafana 的标准监控体系。
